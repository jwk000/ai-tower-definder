## 3. Buff系统

### 3.1 Buff属性

| 属性 | 说明 |
|------|------|
| **增益** | 正面效果（加ATK/HP/速度等） |
| **减益** | 负面效果（减速/减攻/眩晕等） |
| **可影响属性** | HP、ATK、Speed、Defense(护甲)、魔抗、Range、AttackSpeed |
| **类别（category）** | 减速类 / 眩晕控制类 / 燃烧 DOT 类 / 增益类 / 标记类 |

### 3.2 Buff规则

#### 基础规则

- **同名 Buff**：刷新持续时间，层数叠加（不超过 `maxStacks`）
- **叠加上限**：每种 Buff 独立设定 `maxStacks` 最大叠加层数
- **永久 Buff**：`duration = -1`，不被计时清除（如建造增益、地形增益）
- **来源追踪**：记录来源实体 ID（`sourceId`），来源死亡时按 §3.2.3 处理
- **过期自动移除**：每帧检查 `duration ≤ 0` 的 Buff 立即移除

#### 3.2.1 全局上限（P1-#8 修复）

为防止 Buff 视觉/逻辑穿屏，引入全局约束：

| 约束 | 值 | 说明 |
|------|-----|------|
| **单实体总 Buff 数量上限** | **8** | 单个实体同时承受最多 8 个不同 Buff（不算层数，算种类） |
| **超出上限时的处理** | LRU 淘汰最早 Buff | 按"最早施加"原则丢弃最旧 Buff，新 Buff 入栈 |
| **特殊保护** | 玩家技能产生的 Buff 不被 LRU 淘汰 | `sourceFaction === Player` 的 Buff 优先保留 |
| **头顶 Buff 图标显示** | 最多 3 个 | 详见 09-ui-ux §6.1。多于 3 个时按优先级（控制 > 减益 > 增益）显示前 3 |

#### 3.2.2 Buff 优先级（多类冲突时）

当多个 Buff 同时影响同一属性时，按以下顺序生效：

| 优先级 | 类别 | 行为 |
|--------|------|------|
| 1（最高） | **眩晕/冰冻控制类** | 完全压制行为（无法移动/攻击），其他 Buff 仍然存在但效果被压制 |
| 2 | **嘲讽控制类** | 强制目标，覆盖一般目标选择 |
| 3 | **减速类** | 同类减速取 **max**（最强生效），跨类减速取 **sum**（详见 [05-combat-system 减速叠加规则](./05-combat-system.md)） |
| 4 | **DOT 燃烧/中毒类** | 每个独立计算伤害，相加但有 MAX_DOT_STACK 上限 |
| 5 | **属性增益（ATK/HP/Speed buff）** | 同类增益取 **max**，跨类乘算（×=1+增益总和） |
| 6（最低） | **标记类**（如"被锁定"） | 不影响数值，仅作为 AI 条件 |

#### 3.2.3 来源死亡清除规则

当 Buff 的来源实体（`sourceId`）死亡时，按以下规则处理：

| Buff 类型 | 来源死亡处理 |
|-----------|--------------|
| **持续效果型**（光环 Buff，如号令塔光环） | **立即移除**——光环消失，Buff 取消 |
| **一次性施加型**（冰塔减速、燃烧 DOT） | **保留**——Buff 已"投出"，与来源无关 |
| **嘲讽**（强制目标某单位） | **立即移除**——目标已不存在 |
| **战吼/Boss 增益** | **保留**直到自然过期——增益已发动 |
| **永久 Buff**（建造增益） | **保留**——来源建筑死亡不会移除已生效永久 Buff |

> 实现侧：Buff 配置中加 `removeOnSourceDeath: boolean` 字段，默认 `false`。

## 5. v3.0 法术卡子分类（追加）

> 自 v3.0 起，玩家主动施放的技能全部以**法术卡**形式存在于卡牌系统中。本节定义法术卡的子分类。机制层面与 §2 预定义技能完全一致，仅是触发方式从"塔的主动技能"改为"手牌出卡"。
>
> 详细数值见 [21-MDA §12](./21-mda-numerical-design.md#12-能量-e-系统数值v30)；法术卡清单见 [03-unit-data §8](./03-unit-data.md#8-卡牌目录v30-新增)。

### 5.1 法术卡按效果分类

| 子分类 | 描述 | 典型代表 | 是否跨波保留 |
|--------|------|---------|------------|
| 即时伤害（AoE） | 在选定区域立即释放范围伤害 | `fireball_spell`, `meteor_spell` | ❌ |
| 持续伤害（DoT/区域） | 在区域内持续若干秒造成伤害 | `arrow_rain_spell` | ❌ |
| 控制（减速/冰冻） | 让敌人减速或暂停 | `slow_spell`, `freeze_all_spell` | ❌ |
| 治疗 | 治疗我方单位或水晶（水晶虽免疫伤害，但治疗法术可恢复其 HP） | `heal_pulse_spell` | ❌ |
| 召唤 | 临时召唤友军单位 | `summon_skeletons_spell` | ❌ |
| 防御（跨波） | 给予水晶（吸收 N 次秒杀的 HP 消耗）或单位临时护盾 | `divine_protection_spell` | ✅ |

### 5.2 法术卡执行接口

法术卡通过 `SpellCastSystem` 执行，与塔主动技能的 `SkillSystem` 共享同一套效果原子：

```ts
type SpellEffect =
  | { type: 'aoe_damage'; center: Vec2; radius: number; damage: number; damageType: DamageType }
  | { type: 'dot_zone'; center: Vec2; radius: number; tickDamage: number; duration: number }
  | { type: 'apply_buff'; targets: 'allies' | 'enemies' | 'allUnits'; buff: BuffSpec }
  | { type: 'summon'; unitId: UnitID; count: number; position: Vec2; lifetime?: number }
  | { type: 'heal'; targets: 'allies' | 'base'; amount: number }
  | { type: 'shield'; targets: 'base' | 'allies'; absorbAmount: number; duration?: number };
```

每张法术卡的 CardConfig 中包含一个 `effects: SpellEffect[]` 数组，由 `SpellCastSystem` 顺序执行。

### 5.3 法术卡能量消耗规则

- **不消耗单位能量**：法术卡从玩家**能量 E 池**扣除（与 §4 单位技能能量分开）。
- **不可与塔主动技能共用**：塔的内置主动技能（如冰塔光环、激光过载）仍走 §4 能量规则，独立于法术卡能量池。
- **数值范围**：法术卡能量消耗 2-10，详见 [21-MDA §12.2](./21-mda-numerical-design.md#122-卡牌能量消耗表)。

### 5.4 法术卡临时升级双轨

法术卡支持两种本局内临时升级（沿用 v3.0 主流程规则）：

| 升级类型 | 作用对象 | 持续时间 | 效果 |
|---------|---------|---------|------|
| 实例临时升级 | 该卡当前在手牌中的实例 | 出卡后消失 | 伤害 +30% / 范围 +10% |
| 卡级临时升级 | 该卡 instanceLevel | 本局 Run 内永久（直到 Run 结束） | 每级累加 +30% 伤害 / +10% 范围 |

> 法术卡死亡（出过一次）后**不回弃牌堆**，与 [25 §1.4 出卡流程](./25-card-roguelike-refactor.md#14-出卡流程) 一致。

## 6. v3.1 塔科技树新增机制（追加）

> 本节为 [30-tower-tech-tree](./30-tower-tech-tree.md) 中各塔终点节点的机制详述，作为 21-MDA 数值表的语义补充。所有具体数值占位以 21-MDA 为准。

### 6.1 贯穿（pierce） — 战术炮塔（炮塔路径 2 终点）

**机制定位**：炮塔路径 2 末端节点（普通炮塔 → 狙击炮塔 → 战术炮塔），承接已废弃的"弩炮塔"穿透概念。

**规则**：
- 战术炮塔发射的**单发弹道**沿初始飞行方向**直线穿透**多个敌人，命中每个目标都结算一次伤害。
- 穿透上限 `maxPierce`：默认 3 个目标，超过后弹道继续飞行但不再造成伤害（视觉上完成抛物线）。
- **穿透衰减**：每穿透一次伤害 × `pierceFalloff`（默认 0.8），即依次为 100% / 80% / 64%。
- 与 AOE 互斥：贯穿弹道**不再触发溅射半径**——这是战术炮塔的取舍（精确穿透 vs 范围杀伤）。
- 与狙击塔的"远距高伤"叠加：贯穿是战术炮塔在狙击塔基础上的**结构性升级**，射程/伤害继承上一节点的数值。

**实现复用**：`ProjectileSystem` 的弹道实体保留 `pierceLeft: number` 字段，命中处理时若 `pierceLeft > 0` 不销毁实体、不触发爆炸，仅扣血并 `pierceLeft -= 1`，命中过的实体 ID 记入 `piercedEids` 防重复打击。

**配置字段**：`techNode.effect.pierce: { maxPierce: 3, falloff: 0.8 }`

### 6.2 传染（infect） — 病毒塔（元素塔路径 3 终点）

**机制定位**：元素塔路径 3 终点（巫毒塔 → 病毒塔），承接已废弃的"毒藤塔 DOT"概念。

**规则（核心传播逻辑）**：

1. **首次施加**：病毒塔普攻命中目标 → 施加「病毒感染」Buff（DOT，5 层上限，5s 持续，每层每秒造成 X 魔法伤害）。
2. **传染触发条件**：被感染目标承受**任何来源的下一次普攻**（不仅是病毒塔自己），触发一次传染判定。
3. **传染目标筛选**：以被感染目标为圆心，半径 `infectRadius`（默认 80px）内查找**未感染**的敌人，按距离从近到远取最多 `infectJumps` 个（默认 3）。
4. **传染衰减**：每跳传染后，新目标的「病毒感染」初始层数 = `floor(源层数 × decayPerHop)`，默认 `decayPerHop = 0.5`。
5. **传染终止条件**：
   - 跳数达到 `infectJumps` 上限
   - 新目标层数衰减到 0（向下取整）
   - 半径内无未感染目标
6. **传染冷却**：每个被感染敌人每 0.5s 最多触发一次传染，防止瞬间链式爆炸。
7. **免疫规则**：Boss 不可被传染感染（可施加首次 Buff 但不作为传染源），与全局 Boss 抗性策略一致。

**与已废弃毒藤塔的关系**：
- 旧毒藤塔的"持续 DOT"语义并入病毒塔的「病毒感染」Buff（dot_infect）。
- 旧毒藤塔的"召唤藤蔓"效果**整体废弃**，不在病毒塔上保留。

**实现复用**：`BuffSystem` 在施加 `dot_infect` 类型 Buff 时挂载 `infectMeta: { jumpsLeft, decayPerHop, infectRadius, lastInfectTime, sourceTowerId }`；每帧检查触发条件，调用 `BuffSystem.applyBuff()` 链式施加。

**配置字段**：`techNode.effect.infect: { radius: 80, jumps: 3, decayPerHop: 0.5, cooldownPerEnemy: 0.5 }`

### 6.3 全屏闪电（chain_global） — 闪电塔（电塔路径终点）

**机制定位**：电塔单路径终点（普通电塔 → 热电塔 → 核电塔 → 闪电塔），核心卖点是**概率触发的全屏 AOE**。

**规则（双阶段触发）**：

**阶段 A — 标记积累**：
- 闪电塔每次普攻（带链式弹跳，第 3 跳后衰减为 0）会给被命中的敌人施加「核电过载」标记（mark Buff，1 层不叠加，10s 过期）。
- 标记不直接造成额外伤害，仅作为阶段 B 的触发素材。

**阶段 B — 全屏闪电释放**：
- **触发概率**：每次闪电塔普攻命中（任意目标）有 `globalChance`（默认 5%）概率激活全屏技。
- **触发独立 CD**：全屏技自身有 `globalCooldown`（默认 10s）冷却，触发后无视该次概率结算，CD 期间不再判定。
- **作用范围**：全屏所有当前承担「核电过载」标记的敌人（不限制视野/距离）。
- **伤害**：每个被命中的敌人受 `globalDamage`（默认 = 该塔普攻基础伤害 × 1.5）的魔法伤害。
- **视觉**：屏幕顶端 → 每个目标头顶各劈一道独立闪电（参考 12-vfx 闪电特效规范）；屏幕短闪白（alpha 0.2，120ms）。
- **音效**：低频隆隆 + 多道叠加的闪电劈裂声（10-audio §15 新增）。
- **触发后状态**：所有被命中敌人的「核电过载」标记**立即清除**（消耗素材）。

**与基础链式攻击的关系**：
- 闪电塔仍保留 3 跳链式（继承核电塔节点），全屏技是**在链式之上叠加的额外能力**，不是替代。
- 全屏技不会再次触发链式弹跳（避免无限链）。

**实现复用**：
- `AttackSystem` 处理闪电塔普攻时，若 RNG 命中 `globalChance` 且 CD ≤ 0，调用 `RuleHandlers.trigger_global_chain` 规则。
- `BuffSystem` 维护 `mark`（核电过载）Buff 列表，handler 查询所有带该 Buff 的实体并独立结算。

**配置字段**：`techNode.effect.chainGlobal: { chance: 0.05, cooldown: 10, damageMult: 1.5 }`

### 6.4 蓄能聚焦（capacitor） — 稳压/特变激光塔（激光塔路径 2）

**机制定位**：激光塔路径 2（普通激光塔 → 稳压激光塔 → 特变激光塔），核心特征是**持续锁定同目标越久伤害越高**。

**规则**：
- 激光塔的攻击模型不同于弹道塔——它持续输出一束激光对单目标造成 DPS（详见 03-unit-data §1）。
- 当激光塔**连续锁定同一个目标**时间超过 `tickInterval`（默认 0.5s），给自身施加一层「激光蓄能」Buff（self_buff 类型，capacitor category）。
- 每层「激光蓄能」给自身 +`stackDamageBonus`%（默认 +15%）总伤害加成，最多 `maxStacks` 层（稳压：6 层 / 特变：10 层）。
- **目标切换重置**：若激光塔切换攻击目标（无论是因为前目标死亡还是范围外脱锁），「激光蓄能」Buff **立即清零**。
- **重新锁定不继承**：即使切换回前目标，蓄能从 0 重新累积。
- **目标死亡补偿**（仅特变）：特变节点提供"目标击杀时保留 50% 蓄能层数"特性（向下取整），鼓励主动选择高血怪。

**Buff 归属**：「激光蓄能」是**塔自身**的 Buff（self_buff），不施加在目标身上，因此不与敌人 Buff 上限（§3.2.1 全局 8 个）冲突。

**与路径 1（多束激光）的关系**：
- 路径 1 强化"广度"（同时打多目标），路径 2 强化"深度"（专精单目标）。
- 互斥不可叠加，符合科技树路径互斥原则。

**实现复用**：
- `AttackSystem` 在激光塔锁定逻辑中追踪 `lockTargetId` + `lockDurationSec`，每 `tickInterval` 调用 `BuffSystem.applyBuff(self, 'laser_capacitor', stacks=1)`。
- `AttackSystem` 处理目标切换时直接清空自身 capacitor 类 Buff。

**配置字段**：`techNode.effect.capacitor: { tickInterval: 0.5, stackBonusPct: 15, maxStacks: 6, killRetainPct: 0.5 }`

## 二、关键决策记录

### 2.1 为什么 8 关主题选这 8 个？

调研显示用户最有记忆点的塔防主题（含 KR 系列、PvZ、BTD、OMD、IM）有以下高频元素：

- **田园/边境**（必须有，教学关，KR/IM/BTD 都用过）→ 关 1
- **沙漠+虫族**（用户给定）→ 关 2
- **冰雪极地**（KR Frontiers Vasunda / Frozen Nightmare 高赞）→ 关 3
- **丛林遗迹/神庙**（OMD2 / KR Forgotten Sands 高赞）→ 关 4
- **港口/海洋**（KR Vengeance Pirate DLC 高赞 + 调研中"海陆混合"几乎无人尝试，有差异化机会）→ 关 5
- **蒸汽朋克工厂**（KR Vengeance Cogforge / BTD6 Workshop 高赞，提供风格切换）→ 关 6
- **孢子菌林**（PvZ Funguy / KR Origins Mactans 概念，调研中"被低估的好机制"）→ 关 7
- **克苏鲁/异界**（KR1 终战 Demon Lord / IM Nexus / Hadestown，作为终战压轴）→ 关 8

我没问用户"选哪 8 个"，因为这是实现细节（AGENTS.md 沟通铁律），用户只关心"风格混搭+三不重复"这条产品诉求。

### 2.2 为什么用 9 字段数值签名而不是直接给数字？

AGENTS.md 第一权威规则："50-mda 是全部游戏数值的唯一权威来源"。如果本文档持有数字，就会和 50-mda 冲突。所以采取折中：

- 本文档持有"相对档位"（极低/低/中/高/极高）—— 设计验收用，描述每关的"体验签名"。
- 50-mda 持有"绝对数值"（HP=100、攻速=1.5/s）—— 实施时填具体数。
- 两者契约：50-mda 在填具体数时，必须让"关 X 的实际平均 HP 落在本文档所述的档位上"。

这样既能验收"数值体验不重复"（用 9 字段表），又不破坏 50-mda 唯一权威。

### 2.3 为什么关 1 不引入全新敌人？

教学关的核心目标是"让玩家学会塔防基础（建塔/能量/出兵/水晶）"，不是"展示新敌人花活"。关 1 沿用现有 `grunt / runner / heavy / boss_commander` 重新着色为 `goblin_scout / wolf_rider / orc_brute / goblin_chief`，做到：

- 现有美术资产复用 → 减少美术成本
- 玩家学习曲线平缓 → 第一关不被任何"新机制"绊倒
- 签名机制只保留 1 个（"战斗中破甲"）→ 单一新概念入门
- 8 波无 Boss → 不引入"Boss 战"概念，留到关 4 mini-boss 才教学

剩余 7 关 + 关 8 终战混编承载所有新内容：**36 全新敌人 + 6 全新 Boss（含 mini）**，平均每关 5 新单位 + 1 新 Boss。

### 2.4 为什么关 3 ↔ 关 4 和关 5 ↔ 关 6 数值轴差异紧贴下限？

设计完后做交叉校验时发现：

- 关 3 暴雪 SIEGE（HP 高、密度 0.5）→ 关 4 神庙 ELITE-SPIKE（HP 中、密度 0.5）：密度同档，HP 只差 1 档。
- 关 5 港口（密度 1.5、节奏 Sustained→Wall）→ 关 6 工厂 GAUNTLET（密度 1.5、节奏 Wall）：密度和节奏都接近。

但这是有意为之的"软切换"：

- 关 3→4 都是"少而强"，玩家的心智从"扛 SIEGE"过渡到"识别精英"，体验是渐变的；
- 关 5→6 都是"持续高强度"，玩家的心智从"双重压力"过渡到"耐力赛"，体验是连续的。

为了避免落到 ≤4 字段差异，校验表里明确标注"实施时优先在 **抗性分布的细节比例** 上把档位差异拉得更明显"——这是给 50-mda 填具体数时的指令。

# 三类单位（陷阱 / 法术 / 场景中立）权威设计落地 — 2026-05-15

## 背景

用户提出：塔/士兵/敌人三类单位已较完整设计，但陷阱障碍、法术（技能/Buff）、场景中立单位三类尚未详细设计。要求参考 PvZ / 怪物火车 / 皇室战争 / 保卫萝卜，**"只要有特色的设定，不要鸡肋设定"**。

## 关键决策（6 项问询全部采纳推荐）

1. **陷阱占路径格**（保卫萝卜风格）→ 强制寻路重算，提升策略深度
2. **场景互动机关完全 levelEditor 预置**（不消耗能量）→ 强化关卡主题感
3. **14 张法术全部设计完**（含读条/自损/全局/撤回）→ 完整体系才有 roguelike 乐趣
4. **新增独立 `category: Scene`**（不复用 Neutral）→ 语义清晰：Neutral=可争夺资源，Scene=主题绑定机关/威胁
5. **环境威胁按主题分配**：墓碑=L6/L9、藤蔓=L2/L3、神龛=L9

## 关键设计决策记录（B4c + E2b 是 v3.4 新机制）

### B4c：关卡通关 3 选 1 单位卡抽卡（§2.6）

**为什么需要**：v3.4 单 Run 闭环 + 无 meta 卡池后，玩家在 Run 中卡组**唯一来源**是开局抽 12 张 + 商店购买 + 秘境奖励。但商店每关只能进 1 次（D1），秘境随机不一定给卡，开局 12 张可能流派性不强。如果不给一个**确定性**的扩卡通道，玩家会感到卡组成长缓慢。

**机制**：每关通关时（终战 W23 例外），强制弹出 3 选 1 单位卡抽卡面板。池子 = 当前未在卡组内的所有单位卡按稀有度权重抽 3 张。选 1 直接加入卡组，跳过不消耗资源。

**与商店购买的差异**：商店是 N 选购买（消耗金币），抽卡是 3 选 1 免费扩组。两者互补：商店保品质（玩家挑想要的），抽卡保数量（每关固定 +1 张）。

**Run 总扩组上限**：8 关 × 最多 1 张 = 8 张（开局 12 + 抽卡 8 = 20 张），加上商店 N 张则可达 25-30 张。

### E2b：秘境 30% 高风险事件（§3.3）

**为什么需要**：v3.4 单 Run 闭环下，"碎片奖励"被替换为"金币+技能点+单位卡混合奖励"，**纯收益型秘境**会让玩家"无脑选秘境"，秘境变成"免费抽奖站"——这与 v1.0 设计"秘境 = 冒险博弈"哲学相反。

**机制**：在 [27-traps-spells-scene §5 秘境事件池](../20-units/27-traps-spells-scene.md#5-秘境事件池) 中约 30% 事件为"高风险类"，可能造成**损 HP / 删卡 / 减能量上限**等伤害性代价。事件呈现时用色调区分（红边框），强警示玩家。

**与商店的张力对比**：商店是稳定收益（明码标价），秘境是高方差（最佳 > 商店，最差 < 商店）。两者博弈给玩家关后决策核心张力。

## 关键设计决策记录

### 数值锚点最终确定（基于 48 §5.4 / §13 关卡金币流量校准）

**1 SP = 50 G 兑换率的设计逻辑**：
- 关后商店金币预算 80-150 G（关 1）→ 700-1000 G（关 8）
- 单 SP 50 G 价值 = "1 个普通卡的价值"或"1.6 个能量瓶价值"
- 单商店 3 SP 上限 = 150 G/商店，约关后预算的 15-100%（避免后期金币爆量直接 dump 成 SP）
- 全 Run 极限金币换 SP = 24 SP（占总流量 80-130 SP 的 20-30%），辅流量而非主流量

**关 N × 2 SP 通关公式**：
- 关 1-8 线性递增（2 → 16 SP），累计 72 SP
- 终战额外加成 20 SP（非 18，突出终极感）
- 92 SP 通关基础 + 30-50 SP 秘境 + 5-15 SP 兑换 = 130 SP 上限
- 节点满级 1 塔（路径 3 节点）= 19 SP（普通塔）或 34 SP（电塔 4 节点）
- → 玩家可"满 1 塔 + 半 1 塔"，符合「Run 内实质性升级 1-2 个塔关键节点」目标

**秘境 SP 奖励池档位（5/15/30/50）**：
- 与关卡深度匹配（关 1-3=5SP / 关 4-6=15SP / 关 7-8=30SP / 隐藏 50SP）
- 权重 40/35/20/5 反映 power curve（早期玩家更多小奖励，后期偶现大奖）

### 商店 8 槽结构哲学

**前 4 单位 + 后 4 功能**的设计：
- 前 4 抽样（不重复 + 卡组排除）→ 流派构筑（玩家选择哪些卡入组）
- 后 4 固定（4 种功能卡）→ Run 维护（SP/能量/HP/手牌）
- 前 4 价格梯度 30/60/120/240 → 4 倍跨度，强差异化
- 后 4 价格 30-80 G → 同档可比，每个功能卡都"承担得起"

**指数刷新成本（30→60→120 G + 3 次硬上限）**：
- v3.0 线性递增 30G × N 在关 6+ 金币爆量下无效（200 G 刷无数次）
- 翻倍后第 3 次刷新 = 30+60+120 = 210 G，相当于"关键 1 张稀有卡"代价
- 3 次硬上限锁死，玩家被迫接受随机或转向秘境

### 经验沉淀

1. **一致性扫描应执行真实 grep 命令而非伪命令**：v0.13 章节的扫描结果块用了伪命令 `rtk grep`，便于阅读但缺乏可重复验证性。v0.14 用实际 grep 输出数据建立统一报表，便于后续 CI 自动化复用（建议第 4 轮代码改造阶段加入 `npm run docs:lint:spcost` / `docs:lint:forbidden-words` 之类的脚本）。

## 一、今日完成

### Phase A3 集成层（today）

把 Phase A2 的四个独立管理器（Energy / Deck / Hand / RunManager）串成一个可被战斗循环调度的整体，并接入 World + main + WaveSystem。

#### A3-1 RunContext 契约枢纽（commit `5b6d043`）

新增 `src/unit-system/RunContext.ts` + 18 单元测试。核心 API：

- `createRunContext({ seed, registry?, level? })` —— 一站式装配 4 管理器 + 6 流 PRNG + 自动调用 `initRun`。容错：`registry` 缺省取全局 `cardConfigRegistry`，空 registry 时 deck 留空不抛错。
- `startWaveEffect(ctx)` —— 波次开始：恢复能量至 `cap`、抽牌补满手牌至 `capacity`。
- `endWaveEffect(ctx)` —— 波次结束：按每张卡的 `persistAcrossWaves` 决定弃手牌。
- `playCard(ctx, instanceId)` —— 出卡三步事务：手牌检查 → 能量检查 → 扣能量 + 离手 + spell 入弃牌堆 / unit 不入。

设计契约：所有出卡入口（UI 拖卡、SpellCastSystem、调试一键出卡）统一走 `playCard`，事务原子性由 RunContext 内部保证。

#### A3-2 World 接入（commit `b7a56d7`）

`src/core/World.ts` 新增：

- `runContext: RunContext | null` 字段
- `attachRunContext(ctx)` / `detachRunContext()` API
- `reset()` 清空 runContext（关卡切换时使用）

设计取舍：runContext 挂在 World 上而非全局单例，原因是测试隔离 —— 每个 World 实例独立持有自己的 Run 状态，避免 vitest 并发用例污染。

#### A3-3 main.initBattle 装配（commit `dd6755d` 同上）

`src/main.ts initBattle()`：

```ts
const runSeed = generateSeed();
initGlobalRandom(runSeed);
// ...
this.world.attachRunContext(createRunContext({ seed: runSeed }));
```

设计取舍：`runSeed` 一份种子双路使用 —— 既喂给全局 PRNG（兼容旧系统），也喂给 RunContext 的 6 流（新系统）。保证整局 Run 完全可复现。

#### A3-4 WaveSystem 波次钩子（commit `dd6755d`）

`src/systems/WaveSystem.ts` ctor 新增 `onWaveStart?: () => void` 可选参数。在 `setPhase(GamePhase.Battle)` 之后立即触发（重要：必须在 Battle phase 设置后，否则 UI 状态机错乱）。

main.ts 注入两个回调：

- **波首**：`startWaveEffect(world.runContext)` —— 补能量 + 满手牌
- **波末**：`endWaveEffect(world.runContext)` —— 按 persist 标志弃手牌
  - 调度顺序：`weather.onWaveEnd()` → `endWaveEffect` → `saveCurrentBattle('wave-end')`
  - 设计理由：存档必须保存"弃完手牌后的状态"，符合 13-save §6 一致性要求

容错：`world.runContext` 为 null 时跳过卡牌系统调用，旧建造流程不受影响。

### 9.3 关键技术决策

1. **DEV-only 隔离**：编辑器代码完全包在 `if (import.meta.env.DEV)` + 动态 `await import()` 内；Vite plugin `apply: 'serve'` 自动只在 dev 启用。生产构建验证：`grep -c "level-editor-host\|bootstrapEditor\|EditorRoot" dist/assets/index-*.js = 0`，编辑器代码完全被 tree-shake。
2. **安全约束 chokepoint**：所有路径访问统一通过 `resolveLevelPath()` —— 白名单正则 `^[a-z0-9_-]+$` + 长度 ≤64 + `path.resolve` 后强制 `dirname === levelsDir`。深度防御：dispatcher 已校验一次，handler 内仍校验一次。
3. **原子写**：tmp 文件 `${filePath}.${pid}.${ts}.${rand}.tmp` + `fs.rename` POSIX 原子操作，并发安全；异常路径自动 rm 清理。
4. **软删 + 时间戳唯一**：DELETE 不真删，移到 `.editor-trash/<id>.<ISO-stamp>.yaml`，多次删同 id 不冲突。
5. **dup 不覆盖**：POST `/dup` 拦截目标已存在 → 409，保护既有关卡。
6. **Result<T> 模式**：客户端 `LevelEditor` 所有 API 返回 `{ ok, value }` 或 `{ ok: false, error }`，不抛异常。
7. **EventTarget 状态**：`LevelEditor extends EventTarget`，UI 订阅 `'change'` 事件刷新（无需第三方状态库）。
8. **Preact 集成**：`tsconfig.json` 加 `jsx: react-jsx` + `jsxImportSource: preact`；`vite.config.ts` 注册 `@preact/preset-vite`；测试中 mock `preact` 模块避免引入 jsdom。

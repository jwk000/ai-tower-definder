## 蝙蝠塔攻击动作设计

**背景**：蝙蝠塔（Bat Tower）的攻击架构与其他塔完全不同——攻击由独立的蝙蝠群成员（Bat Swarm Member）实体执行。当前 `BatSwarmSystem.tryAttack()` 中仅有数据层面 `applyDamageToTarget()` + 目标 `hitFlashTimer`，蝙蝠自身无任何攻击动画反馈。

**工作内容**：
- 深入分析了蝙蝠塔全链路实现：bitecs 组件定义、建造流程、蝙蝠实体创建、攻击逻辑、boid 集群运动、复合渲染、AI 行为树
- 梳理了现有 7 种攻击动画模式：受击闪白、临时特效实体、持续连线、屏幕震动、状态效果、陷阱伸缩、蝙蝠翼拍悬浮
- 设计并写入四阶段攻击动作方案到设计文档

**设计决策**：

| 阶段 | 持续时间 | 核心视觉 |
|------|---------|---------|
| 锁定蓄力 | ~0.27s (20%) | 翼展加大、暗影粒子凝聚、转身对准目标 |
| 俯冲攻击 | ~0.53s (40%) | 贝塞尔弧线飞行、高频拍翅、暗影拖尾粒子 |
| 撕咬命中 | ~0.20s (15%) | 红色獠牙绘制、命中粒子爆散、绿光吸血回流、屏幕微震 |
| 回弹归巢 | ~0.33s (25%) | 弹回 boid 位置、翅膀减速、绿色吸血余光淡出 |

关键设计点：
- 伤害延迟到 phase=0.65（撕咬瞬间）而非攻击启动即刻——让玩家看到因果链
- 攻击期覆盖 boid 位置计算，避免集群算法干扰攻击轨迹
- 弧高 15-30px 随机，让多只蝙蝠同时攻击时轨迹不重叠
- 复用现有 ParticleContainer / BloodParticleSystem 粒子池，不引入新渲染系统

**修改文件**：
- `design/12-visual-effects.md` — 新增 §9 蝙蝠塔攻击动作设计（~200行）
- `design/16-art-assets-design.md` — 更新攻击动画表蝙蝠塔行，引用新 §9

## 验收修复：基地不掉血 + 起始金币对齐

**背景**：用户验收反馈两个问题——「开局金币太少了无法安放任何单位」、「小兵攻击基地基地不掉血」。

**问题 1 诊断（基地不掉血）**：
深入排查 MovementSystem.onReachEnd 中读取 `Attack.damage[eid] ?? 10` 永远取 0 的现象。根因是 bitecs `Types.f32` TypedArray 的固有行为——未对 entity addComponent(Attack) 时，TypedArray 默认值为 0 而非 undefined，导致空值合并 `??` 永远不触发回退。Grunt 等近战敌人（`attackRange=0` 不挂 Attack 组件）走到终点造成 0 伤害。

**问题 2 诊断（金币不足）**：
对照 `design/21-mda-numerical-design.md` §7.1 起始金币表，发现 4 个关卡配置与权威表不一致：L1 少 20G、L2 少 10G、L4 多 10G、L5 多 20G。

**修复方案**（TDD 流程）：

| 阶段 | 工作 |
|------|------|
| 红 | 先写 7 个 MovementSystem 回归测试：近战敌人扣血、远程敌人扣血、不同 atk 值、Boss、自伤保护、destroy 时机、无基地早返。运行确认 5/7 红。 |
| 绿 | UnitTag 新增 `atk: Types.f32` 字段，WaveSystem.spawnEnemy 时统一写入 `config.atk`，MovementSystem.onReachEnd 改读 `UnitTag.atk[eid]`。再跑测试 7/7 全绿。 |
| 对齐 | 修改 level-01/02/04/05.ts 起始金币至文档值（220/260/340/380）；同步修正 design/07 示例值。 |
| 回归 | `npm run typecheck` 通过；`npm test` 全量 366/366 绿（基线 359 + 新增 7）。 |

**关键设计决策**：
- 选择 UnitTag.atk 作为伤害唯一来源（而非新增专用组件或保留 Attack 双源），理由：所有敌人都有 UnitTag，数据契约统一；spawn 时一次写入，避免 onReachEnd 路径再 lookup `ENEMY_CONFIGS[type].atk` 的运行时开销。
- 在 MovementSystem.onReachEnd 留下 docstring 警告，明确告知未来重构者「不要回退到 `Attack.damage[eid] ?? 0`」——bitecs TypedArray 默认 0 是已知陷阱，避免回归。

**修改文件**（两个原子提交）：
- 提交 1 `fix(combat)`：`src/core/components.ts`、`src/systems/WaveSystem.ts`、`src/systems/MovementSystem.ts`、`src/systems/MovementSystem.test.ts`（新）。
- 提交 2 `chore(balance)`：`src/data/levels/level-{01,02,04,05}.ts`、`design/07-map-level-system.md`。

## 验收修复：关卡开局金币被残留 BattleSnapshot 覆盖

**背景**：用户验收反馈「关卡开局配置金币不是 10G，但实际进入游戏是 10G」。L1 配置 `startingGold: 220`，实际显示却是 10。

**根因定位**：
1. `main.ts:332` 正确设置 `this.economy.gold = config.startingGold` (=220)。
2. 紧接着 `main.ts:736 tryRestoreBattleSnapshot()` 加载 localStorage 中的 BattleSnapshot——只要 `snapshot.levelId === currentLevelId` 就用 `snapshot.economy.gold` 覆盖。
3. SaveManager 只在胜利/失败时 `clearBattleSnapshot()`，**中途从战斗界面返回关卡选择没有清** ——SaveManager.ts:262 注释明确写了应在「victory/defeat/abandon」时清除，abandon 路径是个遗漏。
4. 复现路径：上一会话进 L1 → 玩到金币剩 10 → 退到关卡选择（snapshot 未清）→ 重选 L1 进入 → 配置 220 被残留 10 覆盖。

**设计权衡**：
design/13 §2 表格只写了「页面卸载时保存」用于意外恢复，但**没明说主动重选关卡如何处理 snapshot**。两个边界场景必须同时满足：
- 用户主动从战斗界面返回关卡选择 → 视为放弃当前战斗，**应清 snapshot**
- 浏览器崩溃/刷新（beforeunload）后再次打开游戏 → **必须保留 snapshot** 以实现意外恢复

因此在 `enterLevelSelect()` 中**仅当 `currentScreen === GameScreen.Battle` 时**清除 snapshot——首次进入（构造函数调用，currentScreen 默认 LevelSelect）不清，保留意外恢复能力。

**修改文件**：
- `src/main.ts` — `enterLevelSelect()` 增加条件化 `SaveManager.clearBattleSnapshot()` 调用。
- `src/utils/SaveManager.test.ts` — 新增 `describe('abandon 路径契约')` 两条用例：abandon 后 snapshot=null；beforeunload 后未 abandon 仍可恢复。

**验证**：`npm run typecheck` 通过；`npm test` 14 个测试文件 388 个用例全绿。

**提交**：`4e6c432` docs(dev-log) — P3 R5 落地记录已纳入；本次 R6 smoke 记录可一并续记。

## P4 — BT 攻击节点真接管（6 非-missile 塔 + 3 敌人）

### 背景

P3 R5 已完成导弹塔（tower_missile）的 BT 三节点真接管（select_missile_target → charge_attack → launch_missile_projectile）。但 design/23 §1.2 问题⑦（AttackSystem 覆盖 BT，攻击节点是死代码）只解决了 missile 一支，其余 **6 非-missile 塔**（basic/cannon/ice/bat/lightning/laser）+ **3 敌人**（basic/boss/ranged）的 BT attack 节点仍是空跑——AttackSystem.update + EnemyAttackSystem.update 每帧独立硬编码处理它们的攻击。

P4 目标：参照 P3 R5 模式，将这 9 个单位的 BT 配置升级到 v2.0，新增对应的真接管节点，将 AttackSystem/EnemyAttackSystem 薄化为「工具库 + cooldown tick」。

### 设计决策

**用户选了能力节点路线 A**（推荐方案）：
- 节点粒度按「能力」分类，避免单位独占节点导致爆炸：
  - `spawn_projectile_tower` —— 服务 basic/cannon/ice/bat 4 个常规弹道塔
  - `lightning_chain` —— 服务 lightning 塔（链式电击 + LightningBolt 视觉）
  - `laser_beam` —— 服务 laser 塔（多束扫描 + LaserBeam 视觉）
  - `enemy_melee_attack` —— 服务 enemy_basic / enemy_boss 近战 + enemy_ranged 兜底
  - `enemy_ranged_attack` —— 服务 enemy_ranged 远程

**cooldown tick 路由**（用户选保留模式）：AttackSystem cooldown tick 全部交给 AISystem（line 130-133，每帧 dt 递减到 0 ceil），BT 节点 set 满 cooldown，AttackSystem.update / EnemyAttackSystem.update 不再守卫 cooldown。

**laser_beam 节点例外**：laser 塔 BT 不读 current_target（节点自扫多束），TOWER_LASER_AI v2.0 删除 check_enemy_in_range 前置——节点自管扫敌+多束分配。

**R5+R6 / R7 合并提交**（用户选）：单切 AI 配置会与 AttackSystem.update 残留逻辑造成同一帧双重攻击（节点 spawn + AttackSystem 又 spawn）。参照 P3 R5（b7217ef）模式，AI 配置切 v2.0 + System.update 薄化必须原子合并提交，避免中间态。

### 节点契约（重点）

5 节点统一行为模式（向 P3 R5 missile 三节点看齐）：

| 节点 | 守卫失败语义 | 成功语义 |
|------|--------------|----------|
| `spawn_projectile_tower` | 无 target / dead / 超 range / cooldown > 0 → FAILURE | spawnProjectile + sound + cooldown reset → SUCCESS |
| `lightning_chain` | 无 target / dead / cooldown > 0 → FAILURE | doLightningAttack（首敌伤害 + 链跳 LightningBolt） + cooldown → SUCCESS |
| `laser_beam` | 无 enemy in range / cooldown > 0 → FAILURE | findEnemiesInRange + 截 getLaserBeamCount + doLaserAttack（伤害 + LaserBeam）+ cooldown → SUCCESS |
| `enemy_melee_attack` | 无 target / dead / 超 range → FAILURE + 清 targetId + MoveMode=FollowPath；cooldown > 0 → FAILURE 但保 HoldPosition | doEnemyAttack（Physical 直伤 + sound）+ HoldPosition + cooldown → SUCCESS |
| `enemy_ranged_attack` | 同上 melee | doEnemyAttack（spawn 红色 Circle projectile 200 px/s + sound）+ HoldPosition + cooldown → SUCCESS |

**关键差异**：敌人节点失败时**主动恢复 FollowPath**（避免攻击中断后僵在原地），塔节点失败不动 movement（塔无 movement）。

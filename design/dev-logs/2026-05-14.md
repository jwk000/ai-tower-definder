用户原始诉求要点：

- 关内不再升级，升级移到关卡外卡池界面。
- 升级不是线性，而是科技树（多路径），每塔 1-3 条路径。

我评估并回复 6 个关键设计点：

1. **基础节点是否共享** —— 用户同意「单基础塔卡 + 两路径分支」，避免卡池爆炸。
2. **关内临时升级是否保留** —— 用户决定保留 `instanceLevel`（仅本局、塔死亡清零、不带出关外）。
3. **路径切换粒度** —— 用户同意"关外可切换，关内不可"。
4. **弩炮塔穿透归宿** —— 用户决定承接到炮塔路径 2 终点（战术炮塔 = 狙击 + 穿透，去掉眩晕 Boss）。
5. **真火塔能量丹** —— 用户同意复用 `EnergySystem`（击杀概率 +1 E，不新增资源）。
6. **导弹分裂语义** —— 用户改为"同地格多发射，单发伤害↓总伤害↑"。

## 二、关键设计决策记录

| # | 决策 | 拒绝方案 | 采纳理由 |
|---|------|---------|---------|
| 1 | 单基础塔卡 + 两路径分支可点满 + 单局装备一条 | 每条路径单独的"路径卡" | 避免卡池种类爆炸（7 塔 × 2 路径 = 14 张卡 vs 7 张基础卡） |
| 2 | 关内保留 instanceLevel（仅数值不切形态） | 关内完全禁升级 | 复用现有 v3 机制；保留法术卡"狂怀术"等关内决策深度 |
| 3 | 真火塔能量丹复用 EnergySystem +1 E | 引入新资源"能量丹" | 不破坏 v3.0 三资源轴；UI/经济链不动 |
| 4 | 弩炮塔穿透归到战术炮塔 | 直接彻底删除穿透机制 | 保留"阵列敌"解法，避免玩家工具箱缩水 |
| 5 | 路径重置返还 70% | 50% / 100% | 70% 略宽松，鼓励试错；100% 会让"路径选择"失去意义 |
| 6 | 旧 baseLevel=N 迁移为路径 1 前 N 节点解锁 | 重置玩家进度 + 全额补偿碎片 | 平滑迁移、不破坏老玩家投入感、不需要额外碎片预算 |
| 7 | 闪电塔全屏技 5% 概率 + 10s CD | 主动技能（玩家手动触发） | 与塔防"放置类"主线一致；避免引入塔的技能 UI |

### 6.2 阶段里程碑

**核心成果**：战斗系统三大消费者（WaveSystem / MovementSystem / MissileTargeting）已全部脱离对 `map.enemyPath` 的直接依赖，改读 `resolveGraphFromMap(map)` 输出的 `ResolvedGraph`：

- **WaveSystem**：`spawnEnemy` 走 `resolvedSpawns` 拿 spawn 坐标（B.11）
- **MovementSystem**：构造器内 `linearizeForLegacy` 把图退化成单链 `GridPos[]`，update 循环改读 `this.path`（B.12a）—— 极小切片，外部行为零变化
- **MissileTargeting**：`evaluateMissileTarget` 通过 WeakMap 缓存 `ResolvedGraph`，取 `crystal_anchor` 角色节点替代 `enemyPath[length-1]`（B.13）
- **5 个 level fixture**：模块顶层调用 `migrateEnemyPathToGraph` 静态派生 `pathGraph + spawns`，loaderAdapter 命中 fixture 已派生字段而非每次 migrate（B.14）

**字段双轨状态**：`MapConfig.enemyPath` 仍存活，因为：
- 类型层 required
- 5 个 yaml fixture 仍是 `enemyPath` 编码
- migration / loaderAdapter 内部 fallback 仍读 enemyPath
- 5 个 ts fixture dual-track（同时有 enemyPath 和 pathGraph）

下次工作（B.15）做最后清场：删 MapConfig.enemyPath、迁移 5 yaml、清 ts fixture 的 enemyPath，进入纯图模型。

### 6.3 关键设计决策

**B.12 拆分**：原 B.12 是 MovementSystem 真图遍历，工程量大、风险高。与用户确认后拆为：
- **B.12a 极小切片**：MovementSystem 内部 path 来源从 `this.map.enemyPath` 换成 `linearizeForLegacy(resolveGraphFromMap(map))`，单链场景下输出严格等价。仅触动构造器 + 三处 `this.path` 替换，update 循环 282 行不动。零风险落地。
- **B.12b 真图遍历**：Movement 组件 schema 改造（pathIndex 重语义为 graph.nodes 索引）、随机分支、portal 跳转、crystal_anchor 检测。**推迟到游戏关卡确有多分支/传送门需求时再做**——当前所有关卡都是单链路径，B.12a 完全够用。

**B.13 WeakMap 缓存**：MissileTargeting.evaluateMissileTarget 每帧多次调用，每次 build graph 浪费。采用 `WeakMap<MapConfig, ResolvedGraph | null>` 缓存，相同 map 引用复用 graph。`getCachedGraph` try/catch fallback null —— BehaviorTree.test.ts 的 13 个 SelectMissileTargetNode 测试用无 enemyPath fixture，需优雅 fallback 到 `MAP_ROWS-1` / `MAP_COLS-1`（与旧代码兜底等价），保留 156 测试全绿。

**B.14 零行为漂移设计**：fixture 不硬编码 graph，而是模块顶层 `migrateEnemyPathToGraph({ enemyPath: WAYPOINTS })` 派生。每次模块加载 migrate 跑一次（开销可忽略），保证 fixture 的 graph **永远等于 migration 输出**，零 mismatch 风险。TDD 测试用 `expect(map.pathGraph).toEqual(derived.pathGraph)` 锁死契约。

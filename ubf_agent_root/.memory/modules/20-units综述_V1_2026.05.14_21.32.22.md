## 核心：单位四级范畴（20-unit-system + 21-unit-roster）

游戏中的所有「实体」可以归到四个范畴，但在 ECS 层面**只有一种 `eid` + 组件组合**，差别完全由配置决定：

| 范畴 | 阵营 | 由谁部署 | 例子 |
|---|---|---|---|
| **Tower（塔）** | player | 玩家拖卡（单位卡） | 箭塔、加农炮、元素塔（冰/火/毒）、导弹塔 |
| **Soldier（士兵）** | player | 玩家拖卡（单位卡） | 矿工、战士、骑士、法师 |
| **Enemy（敌人）** | enemy | 系统按波次生成 | 兽人、骷髅、飞龙、Boss |
| **Neutral（中立）** | neutral / objective | 关卡预置 | 水晶（玩家方目标）、宝箱、机关 |

**阵营关系由 `62-faction-refactor.md` 的 `isHostileTo(a, b)` 决定**，不要用 `isEnemy` 双轨布尔判断。

### v3.0/v3.1 阵容规模

- 敌方：20 个单位（基础 + 精英 + Boss）
- 友方：20 个单位（塔 + 士兵 + 建筑 + 陷阱）
- 卡牌目录：`21-unit-roster §7` 列出完整 `CardConfig ↔ UnitConfig` 映射。

## 塔科技树（22-tower-tech-tree，v3.1 权威）

### v3.1 的核心变更

> v3.0 把塔卡升级保留在**关间商店**（玩家用金币升级）。v3.1 把塔卡升级**单独剥离**出来，转到**关外卡池科技树**（玩家用火花碎片解锁节点）。

**理由**：
- 关内升级让卡组流派太可预测（升满 = 强）。
- 关外科技树 + 路径互斥（如冰塔可走「群冰路径」或「单体爆冰路径」，互斥）让流派构筑变成跨 Run 长线决策。

### 7 塔节点结构

7 个塔（箭塔、加农炮、元素塔、导弹塔、激光塔、闪电塔、矿工塔等，具体见 22 文档原文）每个都有 1-2 条**路径互斥**的科技路线，每条路线有**线性解锁的节点**。

- **解锁货币**：火花碎片。
- **存档字段**：`CardEntry.techTree`（见 `61-save-system.md`）。
- **路径重置**：通过商店「路径重置券」（v0.6 引入）或火花碎片成本重置。

### 关内 vs 关外的边界

| 操作 | 关内（出卡后） | 关外（卡池界面） |
|---|---|---|
| 塔的形态（冰/火/毒） | ⛔ 不可切换 | ✅ 科技树选路径 |
| 塔的 `instanceLevel` 提升 | ✅ 法术卡触发（仅本局） | ⛔ |
| 非塔卡升级（士兵/法术/陷阱） | ✅ 商店购买 | ⛔ |

### 废弃单位（v3.1）

- **毒藤塔 / 弩炮塔** → 完整原设计归档到 `design/archive/deprecated-units-vine-ballista.md`。
- **ice_tower → elemental_tower**：默认冰形态，路径覆盖冰/火/毒。

## 技能与 Buff 系统（23-skill-buff.md）

### 双轨升级（v0.2 起）

技能卡（spell card）有**两个独立的升级维度**：

1. **`instanceLevel`**：通过**法术卡** `spellSubtype='buff_instance'` 在关内对**已部署的单位实例**升级（仅本局有效）。
2. **卡级（cardLevel）**：通过关间商店升级**该卡牌在本局所有未来出场实例**的基础等级（仅本局有效）。⛔ 塔卡的此升级在 v3.1 被禁用。

> 跨局永久升级走的是 `CardEntry.techTree` 而不是 cardLevel。

### Buff/Debuff 运行时

- **作用域**：单位实例（`eid`）级，不是单位类型级。
- **保护规则**（玩家阵营 buff 保护）见 `25-vulnerability.md`：例如 buff_instance 不会被同等优先级的 debuff 覆盖。
- **优先级**：debuff 之间按 priority 排序，详见 25 文档。

### 法术卡 5 个子类（spellSubtype）

| 子类 | 作用 | 例子 |
|---|---|---|
| `damage` | 立即伤害 | 火球术、闪电 |
| `control` | 控制（眩晕/减速/沉默） | 冰冻 |
| `buff_instance` | 升级**已部署单位实例**的 `instanceLevel` | 锻造、强化 |
| `buff_card` | 升级**手牌中的卡**（少见，特殊事件给） | 神圣祝福 |
| `utility` | 工具类（抽卡、能量、移除卡） | 灵感 |

## 战斗公式骨架（24-combat.md）

> ⚠️ 公式骨架在此，**所有数值在 50-mda**。本文档只描述变量关系。

### 伤害管线

```
攻击方 → 选目标 → 判命中 → 计算基础伤害（atk * atkMul）
      → 应用 buff/debuff 修正（attack% + defense% + 状态）
      → 应用护甲（damage * (1 - armorReduction)）
      → 应用脆弱/抗性
      → 扣 HP → 触发 onHit 规则
```

### 攻速 / 移速规范

- 攻速 = `每秒攻击次数`（不是攻击间隔毫秒）。
- 移速 = `每秒像素数`（不是网格 / 秒）。
- 眩晕 = 攻击间隔暂停（攻速冷却不进），不是直接清零冷却。
- 弹道 = `ProjectileSystem` 内独立实体，飞行中可被拦截或穿透。

详见 24 文档与 `60-architecture.md` §3 渲染层。

## 漏洞自检（25-vulnerability.md）

这份文档不是新设计，是**已修复漏洞的自检报告 + 回归测试映射**。每个漏洞条目对应：

- 触发场景（如「玩家 buff 被同等 priority debuff 顶替」）。
- 修复时机。
- 关联的回归测试 case ID（vitest）。
- v3.1 audit 状态。

**实现侧用法**：修改 buff/伤害/控制逻辑时，必须先看 25 文档列出的回归 case 是否仍然 pass。

## 导弹塔（26-missile-special.md）

导弹塔是个**特殊战略武器**，与其它塔行为差别大，单独成文：

- **全图射程**（不像其它塔有 range 圆）。
- **地格评分**：选目标用地格评分（密度 + 价值），不是单体目标选择。
- **大范围爆炸**：AOE 半径远大于其它 AOE 塔。
- **冷却长**：每次发射后 CD 长，逼迫玩家挑节点关键时刻使用。

> 这份文档是「特例文档」的范本：单位行为差别足够大时，在 21-unit-roster 中保留简表，但**完整设计独立成文**。

## 实现侧关键约束

1. **新增单位 = 写 YAML，不改代码**。`stats / behavior / lifecycle / ai / visual` 五段配置完成即可。
2. **塔卡升级有两条路径**：关外科技树（永久）+ 关内法术卡 instanceLevel（仅本局），二者不冲突。
3. **`isHostileTo` 是阵营判断唯一 API**：禁止 `isEnemy` 双轨布尔。
4. **导弹塔目标选择特殊**：不要套用 `ScoreSelectTarget` 通用评分节点，单独实现。
5. **buff/debuff 改动必须看 25-vulnerability 回归映射**：改了之后跑全量 vitest，确认 25 列出的 case 仍 pass。

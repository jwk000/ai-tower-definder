# 30-ai 综述 — AI 行为

> 本层 = **单位行为的统一调度方式**。游戏中所有单位（塔、士兵、敌人、Boss）的目标选择 + 攻击决策 + 移动决策都由**行为树（Behavior Tree, BT）** 配置驱动。

## 设计哲学

### 一句话：行为树是单位行为的统一调度，规则引擎是生命周期事件的统一调度

- **行为树（BT）** 决定单位**主动行为**：选谁打、走哪、什么时候撤退。
- **规则引擎（RuleEngine）** 决定单位**被动事件**：被打了触发什么、死亡触发什么、被治疗触发什么。
- 两者不冲突，互相补充。

### 为什么不是脚本/状态机？

- 早期项目分散在 `AttackSystem` / `MovementSystem` 各种 if-else，新增单位行为要改多处。
- 改成 BT 后，每个单位只声明一个 `ai_tree` 字段 → 行为完全数据驱动 → 策划可编辑。
- 80% 单位只用预设 BT（如 "tower_default"、"enemy_grunt"），20% 复杂单位才写自定义 BT。

> 早期评估时认为「AI 系统过度设计」（80% 单位只需简单 AI），但在 v3.1 阵容扩展到 40 个单位后这个判断被修正：BT 的可配置性反而成为新增单位的速度优势。

## 行为树运行时（30-behavior-tree.md）

### BT 节点目录

行为树的节点分四类（具体节点见 30 文档原文）：

| 类型 | 作用 | 例子 |
|---|---|---|
| **Composite（组合节点）** | 控制子节点执行顺序 | Sequence、Selector、Parallel |
| **Decorator（装饰节点）** | 修改子节点结果 | Inverter、Repeater、Cooldown |
| **Action（动作节点）** | 实际做事 | MoveTo、Attack、SelectTarget |
| **Condition（条件节点）** | 判断状态 | IsTargetInRange、HasHpBelow |

### 核心：ScoreSelectTarget 评分节点

> 目标选择是 AI 中最容易陷入复杂度爆炸的环节。30 文档用 **ScoreSelectTarget 评分节点**统一收敛。

工作方式：
1. 收集候选目标（射程内所有敌方单位）。
2. 对每个候选用配置的 `scoreFormula` 计算分数（如 `hp * 2 + distance * -1 + isFlying * 5`）。
3. 选最高分目标。

**优势**：不同单位用不同 `scoreFormula` 就能表现千差万别的"性格"，不写代码。

**例外**：导弹塔的目标选择走**地格评分**（不是单位评分），单独实现，见 `20-units/26-missile-special.md`。

### 运行时实例

- 每个单位实体有**一个 BT 实例**，状态保存在组件中（当前节点、tick 计数、黑板）。
- BT 在 `Pipeline` 第 7 阶段 `PHASE_AI` 执行（详见 `60-architecture.md`）。
- BT 决策结果通过修改组件（target、moveTarget、attackCooldown）让其它系统执行。

## 士兵 AI 四状态机（31-soldier-ai.md）

> 士兵（Soldier）是**玩家方主动出击的单位**，行为比塔复杂（塔只待在原地打）。31 文档把士兵 AI 用四状态机 + 三圈模型组织。

### 四状态机

| 状态 | 含义 | 出口 |
|---|---|---|
| **Idle（待命）** | 在出生点附近，没有威胁也没有目标 | 发现敌人 → Pursuit；玩家拖拽 → Move |
| **Pursuit（追击）** | 锁定目标向其移动 | 进入攻击范围 → Engage；目标消失 → Idle |
| **Engage（交战）** | 在攻击范围内攻击 | 目标死亡 → Idle；目标逃出范围 → Pursuit；HP 低 → Retreat |
| **Retreat（撤退）** | HP 低于阈值，向出生点撤退 | 回到出生点 → Idle |

### 三圈模型

每个士兵有**三个同心圆**：

1. **视野圈**（最大）：能看到敌人，从 Idle 转 Pursuit 的范围。
2. **攻击圈**（中）：能发动攻击的范围。
3. **粘附圈**（最小）：在 Engage 中允许追击的最大距离，超出则放弃当前目标回 Idle 或换目标。

### v3.1 audit 注意

- §12「士兵升级机制」**延后到 M5 里程碑实现**，当前版本士兵不升级。
- v3.0 部署改成卡牌后，士兵的出生点 = 玩家落卡位置，不是固定出生点。

## 与规则引擎的协作

行为树 + 规则引擎的协作示例（弓箭手塔击杀敌人）：

```
BT.tick:
  → SelectTarget（选最近敌人）
  → IsTargetInRange?（是）
  → Attack（设置 attackCooldown 与触发 onAttack 事件）

AttackSystem 看到 attackCooldown 触发 → 生成投射物 → 命中目标
HealthSystem 检测目标 HP ≤ 0 → 通知 RuleEngine
RuleEngine.onDeath:
  → 弓箭手 onKill 规则（无）
  → 敌人 onDeath 规则（如「自爆 deal_aoe_damage」「掉金币 spawn_loot」）
```

> BT 只管"我决定打"，命中、死亡、连锁效果是 RuleEngine 的事。

## 实现侧关键约束

1. **每个单位都有 BT**，没有"无 AI"的单位。塔的 BT 通常是 `tower_default`（选最近 → 打）。
2. **新增 BT 节点要在 `ai/BehaviorTree.ts` 注册**，然后才能在 YAML 中引用。
3. **BT 预设放 `ai/presets/`**（如 `boss_commander_ai`、`abyss_lord_ai`），单位配置 `ai_tree: "boss_commander_ai"` 引用。
4. **目标选择优先用 ScoreSelectTarget**，特殊场景（如导弹塔的地格评分）才自定义。
5. **不要在 BT 节点内修改 HP/伤害**，那是 RuleEngine 的职责。BT 只设置 target / moveTarget / 触发动作意图。
6. **BT 决策结果是声明式的**，不是命令式的。例如 BT 输出"打 eid=42"，由 `AttackSystem` 决定能不能打、如何打、怎么生成弹道。

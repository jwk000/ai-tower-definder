## 架构总览（60-architecture.md）

### 技术栈

**PixiJS WebGL + bitecs + TypeScript + Vite**，配置驱动 + 规则引擎 + 行为树。

### 四层架构

```
[配置层 YAML]        策划可编辑：单位 / 关卡 / 卡牌 / 技能
   ↓ ConfigLoader
[规则引擎]           生命周期事件 + 行为规则（声明式规则 → 运行时调度）
   ↓
[ECS World - bitecs] 实体 + 组件（SoA 内存）+ 类型安全 query + 系统
   ↓
[PixiJS 渲染层]      Graphics（几何） + ParticleContainer（粒子） + Container（分层）
```

### bitecs ECS 规则

- **组件**：`defineComponent({ field: Types.f32 })` 定义字段集合，纯数据，SoA 布局。
- **实体**：`addEntity(world)` 返回 `eid: number`。
- **查询**：`defineQuery([CompA, CompB])` 类型推断（AND 语义）。
- **系统**：`{ name: string; update(world, dt): void }`，纯函数处理组件数据。
- **死亡清理**：`destroyEntity(eid)` 标记延迟删除，`World.update()` 末尾统一 `removeEntity`。

### Pipeline 8 阶段（核心约束）

```
1. PHASE_MANAGERS    经济 / 波次 / 天气等独立管理器
2. PHASE_VFX         死亡 / 爆炸 / 闪电 / 激光等视觉计时
3. PHASE_MODIFIERS   Buff / 治疗等状态修改
4. PHASE_GAMEPLAY    移动 / 攻击 / 弹道 / 技能 / 陷阱 / 生产
5. PHASE_LIFECYCLE   生命周期事件分发 + 死亡检测
6. PHASE_CREATION    建造 / 实体新建
7. PHASE_AI          行为树执行
8. PHASE_RENDER      RenderSystem + UISystem（始终最后）
```

> ⚠️ **顺序不可调换**。每个阶段内的系统可以重排，但跨阶段顺序固定。改动 Pipeline 顺序需要在 60-architecture 文档中先评审。

### 规则引擎设计

- **生命周期事件**：`onCreate` / `onDeath` / `onHit` / `onAttack` / `onKill` / `onUpgrade` / `onDestroy` / `onEnter` / `onLeave`。
- **触发流程**：系统检测事件 → `ruleEngine.dispatch(event, entity, context)` → 查找单位配置规则 → 执行 `RuleHandler`（如 `deal_aoe_damage`、`apply_buff`）。
- **行为规则**：`targetSelection` / `attackMode` / `movementMode` 在 AttackSystem / MovementSystem 中提供决策。
- **行为树补充**：单位配置 `ai_tree` 时由 BT 接管目标/攻击决策；生命周期规则仍走规则引擎。两者协作不冲突。

### 与行为树的边界

| 决策类型 | 由谁负责 |
|---|---|
| 选目标、攻击、移动 | 行为树 |
| 受击、死亡、击杀连锁效果、被治疗、被 buff/debuff | 规则引擎 |

详见 `30-ai综述` 中的"与规则引擎的协作"示例。

## 存档系统（61-save-system.md）

### 存档版本：v2.0.0（无迁移代码）

游戏未发布，存档结构可随设计调整。**有重大重构时直接改 schema，不写 migration**。

### 存档结构

```typescript
SaveData {
  version: '2.0.0'
  permanentUpgrades: {            // 跨 Run 永久升级
    baseHpMax: number             // 语义：水晶最大 HP（字段名保留 baseHp 兼容）
    ...
  }
  cardCollection: CardCollection  // 永久卡池
  ongoingRun?: OngoingRun         // 关间存档点（未结束的 Run）
  stats: { ... }                  // 历届最远闯关、总 Run 数等
}

CardCollection {
  entries: Record<cardId, CardEntry>
}

CardEntry {
  cardId: string
  unlocked: boolean
  techTree?: TechTreeState         // 塔卡专用：路径选择 + 已解锁节点
}

OngoingRun {
  currentLevel: number             // 1-8
  deck: CardConfig[]               // 本局 12 张卡
  hand: CardConfig[]               // 当前手牌
  discardPile: CardConfig[]
  baseHp: number                   // 当前水晶 HP
  gold: number                     // 当前金币
  weather: WeatherType
  seed: { master, map, wave, ... } // 6 流 PRNG seed
}
```

### 字段命名兼容

- 「水晶」是 v3.0 语义命名，但代码与存档仍用 `baseHp` / `baseHpMax` / `crystalShatter` 等。
- **UI 文案 / 设计文档 / 玩家可见处用「水晶」**，代码侧保留 `baseHp` 不重命名。

## 阵营语义重构（62-faction-refactor.md）

### 历史问题

早期代码有**双轨布尔**：`isEnemy: bool` + `faction: 'player' | 'enemy' | ...`。两个字段经常不一致，导致 Bug。

### 重构方案

- **删除 `isEnemy` 字段**。
- **统一用 `faction` 字段** + `isHostileTo(a, b): boolean` API 判断敌对关系。
- 阵营关系是**可配置的**：默认 player vs enemy 敌对，可在配置中加 neutral、objective 等阵营。

### 实现侧影响

任何修改 / 判断敌对关系的代码必须用 `isHostileTo` API：

```typescript
// ⛔ 禁止
if (target.isEnemy) { ... }

// ✅ 正确
if (isHostileTo(self, target)) { ... }
```

## 调试系统（63-debug.md）

### 关键功能

- **跨界面调试入口**：在主菜单、关内、关间、商店等所有界面提供调试入口（通过快捷键或隐藏按钮）。
- **一键通关**：跳过当前关或当前 Run 到指定关，便于测试关间节点 / 终战。
- **行为树查看**：实时可视化指定单位的 BT 当前状态、tick 历史。
- **实体检视器**：点击任意实体查看其所有组件值。
- **配置热加载**：修改 YAML 后无需重启即可生效。
- **PRNG seed 注入**：手动指定 6 流 seed，便于回放特定情形。

### 调试入口位置

每个 UI 界面有固定的调试入口位置（通常右上或左下隐藏），详见 63 文档。

## 关卡编辑器（64-level-editor.md，⭐ authoritative）

### 定位

- **DEV-only**：仅开发期可用，不打包到 release。
- **YAML schema 权威**：关卡 YAML 的字段定义以 64 文档为准。

### 核心功能

- **图模型路径编辑**：节点 + 边的可视化编辑，支持单/多路径与分叉。
- **波次/池/难度配置**：每关的波次组合、随机池、难度乘数全字段可编辑。
- **一键试玩**：编辑器内直接进入该关测试，无需打包。
- **Preact UI**：编辑器自身用 Preact 实现，与主游戏的 PixiJS 解耦。

### 数据流

```
编辑器 UI → 修改 config/levels/*.yaml → ConfigLoader 重新加载 → 游戏热刷新
```

### 一键试玩

进入试玩后跳过抽卡阶段，直接给玩家完整卡池（或可在编辑器中指定测试卡组），便于测试关卡平衡而不受卡组随机性干扰。

## 实现侧关键约束

1. **Pipeline 8 阶段顺序绝不可乱**。改动需先在 60-architecture 文档中评审。
2. **新增组件用 `defineComponent`**，禁止挂在已有组件上扩字段（破坏 SoA 布局）。
3. **系统是纯函数**，禁止在系统内 `new Component()` 或保持跨帧状态（状态放组件）。
4. **死亡实体走 `destroyEntity` 延迟删除**，禁止在系统内 `removeEntity`。
5. **阵营判断统一用 `isHostileTo`**，禁止 `isEnemy` 双轨。
6. **存档版本号 v2.0.0 不写迁移**。改 schema 时直接改，文档中说明影响。
7. **6 流 PRNG 共用一个 master seed**，派生独立子 seed 给各流，禁止流之间混用。
8. **关卡 YAML schema 改动以 64-level-editor 为准**。先改文档 schema → 改加载器 → 改现有关卡 → 改编辑器。
9. **导入路径用 `.js` 后缀**（Vite bundler moduleResolution 要求）。
10. **TS 严格模式打开 `noUncheckedIndexedAccess`**，数组下标返回 `T | undefined` 必须处理。
